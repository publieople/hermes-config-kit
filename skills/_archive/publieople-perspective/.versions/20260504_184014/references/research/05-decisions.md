# 05-decisions.md — 决策记录与行动

> 从 Publieople（人民公仆）的原创文章和项目中提取的决策模式
> 调研时间：2026-05-04

## 重大决策记录

### 决策1: 从 Linux 回到 Windows + WSL Arch
- **背景**：尝试多个 Linux 发行版后
- **决策逻辑**：
  - 「Linux很好，但是祂并不适合所有人的日常使用」
  - Windows 上有 Quicker、PowerToys、鼠标手势等实用工具
  - Linux 的兼容性问题需要较多成本才能解决
- **结果**：回到 Windows，用 WSL Arch 满足 Linux 需求
- **思维模型体现**：实用 > 纯粹、两条腿走路、最小成本方案

### 决策2: 发起通识分享企划（2024-09-15）
- **背景**：看到通用知识传播的需求
- **决策逻辑**：
  - 非盈利、纯粹知识分享
  - 团队化运营（8人），不是单打独斗
  - 两周一篇的可持续节奏
- **结果**：20+篇原创内容，成体系的知识库
- **思维模型体现**：知识即公共品、进度优于完美、分层学习

### 决策3: 个人主页 Stage 0 → Stage 1 推倒重来
- **背景**：Stage 0 完成后发现遗留问题太多
- **决策逻辑**：
  - 「趁着项目还不大，我删除了大部分文件」
  - 保留 TODO.md、README.md、.gitignore
  - 用 TODO.md 指导 AI 开发
- **结果**：重建非常顺利
- **思维模型体现**：先加后减、敢于推倒重来、AI 即放大器

### 决策4: 选择 Vibe Coding 作为开发手段
- **背景**：前端基础不牢固
- **决策逻辑**：
  - 「只要明确自己的需求，AI 就能很好的完成工作」
  - 「几乎不用自己写代码」
  - 搭配 Cursor 作为主要开发工具
- **结果**：能独立完成前端项目开发
- **思维模型体现**：AI 即放大器、实践驱动学习

### 决策5: 使用 Notion + NotionNext 搭建博客
- **背景**：需要发表公开文章的平台
- **决策逻辑**：Notion 本身就是知识管理工具，用 NotionNext 直接转为博客，减少迁移成本
- **结果**：Publieople's Blog 上线，32 篇文章
- **思维模型体现**：工具即思维延伸、最小成本方案

## 日常决策模式

### 技术选型原则
1. 先找现成工具（Quicker、Everything、PowerToys）
2. 找不到满意工具 → 尝试自己实现
3. 开源优先，可定制优先
4. 不追求最新最好，追求最适配

### 写作决策
1. 先明确目的（注重过程还是结果）
2. 以用户搜索意图作为选题方向
3. 先写初稿再打磨
4. AI 辅助但不主导

### 分享决策
1. 有价值的知识应该公开
2. 团队协作优于个人
3. 可持续节奏比短期爆发重要
4. 保留诚实边界（「本文仅作为科普」）

## 决策矛盾点

| 矛盾 | 场景 A | 场景 B |
|------|--------|--------|
| 技术纯正 vs 实用主义 | 推崇命令行和 Linux | 自己使用 Windows |
| 深度 vs 广度 | 教程覆盖全面但不够深入 | 通识企划定位于广度 |
| AI 信任 vs AI 警惕 | 重度使用 Vibe Coding 和 AI 搜索 | 「不要被 AI 牵着走」「内容需人工审核」 |
